## 3. 数据层详解

### 3.1 五张核心表

| 表 | 行数 | 主键 | 说明 |
|---|------|------|------|
| `stock_list` | 300 | code | 沪深300 成分股清单 |
| `stock_daily` | 74 478 | (code, trade_date) | 日线 OHLCV + 涨跌幅 + 换手率 |
| `stock_feature` | 68 367 | (code, trade_date) | 28 维技术 + 宏观特征 |
| `stock_predict` | 291＋ | (code, trade_date) | iTransformer 三类预测 + 置信度 |
| `stock_report` | 238 | (code, report_date) | DeepSeek 生成的中文研报 |

### 3.2 数据清洗（关键技术细节）

**问题**：早期数据采集时，`stock_list` 和 `stock_daily` 的 `code` 列把深市股票存成了**裸数字**（`'1'` 而不是 `'000001'`），导致前端按 `'000001'` 永远查不到数据。

**修复**：写了 `scripts/cleanup_codes.sql` 一次性脚本。

```sql
-- 补 0 到 6 位
UPDATE stock_list   SET code = LPAD(code, 6, '0') WHERE LENGTH(code)<6 AND code REGEXP '^[0-9]+$';
UPDATE stock_daily  SET code = LPAD(code, 6, '0') WHERE LENGTH(code)<6 AND code REGEXP '^[0-9]+$';
-- 其余表同理
```

**坑 1 ：collation 冲突**。原本想用 `WHERE name LIKE '%上海%'` 顺便规范 `exchange` 列。但 `stock_list.exchange` 是 `gbk_chinese_ci`，而 MySQL 客户端连进来默认 `utf8mb4_general_ci`，跑下去报：

```
Illegal mix of collations (gbk_chinese_ci,IMPLICIT) and (utf8mb4_general_ci,COERCIBLE)
```

**绕开方法**：完全不碰中文，改用 **code 前缀 REGEXP**：

```sql
UPDATE stock_list SET exchange='SH' WHERE code REGEXP '^(6|9)' AND exchange NOT IN ('SH','SZ','BJ');
UPDATE stock_list SET exchange='SZ' WHERE code REGEXP '^(0|3)' AND exchange NOT IN ('SH','SZ','BJ');
UPDATE stock_list SET exchange='BJ' WHERE code REGEXP '^(4|8)' AND exchange NOT IN ('SH','SZ','BJ');
```

跑完后所有 300 行 `code` 都是 6 位字符串、`exchange` 都规范成 `SH / SZ / BJ`。

**坑 2 ：双 candidate 兜底**。即便 DB 清洗了，我们在 `main.py` 里依然保留了**入口卫兵 `normalize_code`** + **WHERE IN `{zfilled, raw_int}` 双查兼容**。这样后续如果有别人再灌进脏数据，接口也不会立刻挂。代价为 0（多一个 IN 列表项），但鲁棒性提高。

### 3.3 时间窗口划分

为了避免任何形式的数据泄漏，严格按时间切：

| 阶段 | 范围 |
|------|------|
| Train | `trade_date ≤ 2026-04-30`（约 60 000 行） |
| Val | `2026-05-01 ≤ trade_date ≤ 2026-05-31`（约 6 000 行） |
| Predict | `trade_date ≥ 2026-06-01`（每日新增） |

- Scaler 仅在 train 上 fit，val / predict 上只 transform。
- 标签是 t+1 涨跌（`pct_chg_{t+1}`），特征是 [t-19, t] 的 20 天窗口，**绝对不允许任何 t+1 信息进入特征侧**。

## 4. 特征工程（28 维）

完整列表（`stock-ai/model/config.py` 中的 `FEATURE_COLS`）：

| 派系 | 字段 | 含义 |
|------|------|------|
| 均线类 | ma5 / ma10 / ma20 / ma60 | 5/10/20/60 日简单均线 |
| 动量类 | macd_dif / macd_dea / macd_bar / rsi | MACD 三件套 + RSI |
| 波动类 | boll_upper / boll_mid / boll_lower / boll_width / atr / volatility | 布林带 + ATR + 波动率 |
| 量能类 | volume_ma5 / volume_ratio / turnover_z | 量均线 + 量比 + 换手 z-score |
| 收益类 | ret_1d / ret_5d / ret_20d | 1/5/20 日收益 |
| 随机指标 | kdj_k / kdj_d / kdj_j | KDJ 三件套 |
| 位置类 | price_position | 60 日内股价相对位置 (0-1) |
| 宏观类 | shibor_on / cpi_yoy / pmi / m2_growth | 流动性 / 通胀 / 经济 / 货币 |

### 4.1 为什么混入宏观（核心创新）

绝大多数学生项目只用 OHLCV + 简单技术指标。我们额外把**宏观 4 维**当作**同期外生变量**塞进每个时间步：

- `shibor_on` — 隔夜银行间利率，反映**流动性**
- `cpi_yoy` — CPI 同比，反映**通胀**
- `pmi` — 制造业 PMI，反映**经济周期**
- `m2_growth` — M2 同比增速，反映**货币宽松度**

这些字段从 `dws_macro_daily` 表里**按日期 join** 进特征矩阵，对所有股票同一天**共享**。

**消融实验**：拿掉宏观四列重训 → val F1-macro 从 36.55% 跌到 ~34.4%。**约 2 pp 的真实提升**。

### 4.2 标签构造

```python
pct_chg_next = (close_{t+1} - close_t) / close_t * 100

label = -1 if pct_chg_next < -1%   else
         0 if -1% ≤ pct_chg_next ≤ +1% else
         1 if pct_chg_next > +1%

LABEL_MAP = {-1: 0, 0: 1, 1: 2}   # 喂给 CrossEntropy
```

阈值取 ±1% 经过实验决定 —— ±0.5% 会让"震荡类"严重不平衡（占 60%+），±2% 让"涨/跌"类太少（< 15%）。±1% 切下来三类大约 33% / 34% / 33%，**天然平衡**。

## 5. 模型选型 —— 为什么是 iTransformer

这是答辩**最重要**的一节，也是评委最可能追问的一节。

### 5.1 候选模型对比

| 模型 | 提出年份 | 核心思想 | 是否选用 | 理由 |
|------|----------|----------|----------|------|
| **LSTM** | 1997 | 门控循环网络 | ❌ | 长依赖梯度衰减；无显式变量间建模；训练慢 |
| **Vanilla Transformer** | 2017 | 时间步 self-attention | ❌ | 短窗 + 多变量场景下 token 数太少，O(L²) 用不满 |
| **Informer** | 2021 AAAI | ProbSparse attention | ❌ | 为超长序列（>500）设计，我们窗口只有 20 天，杀鸡用牛刀 |
| **PatchTST** | 2023 ICLR | 时序切 patch、变量独立 | ❌ | 变量独立 → 学不到 MACD↔KDJ↔SHIBOR 的耦合 |
| **TFT** | 2021 | Gated residual + multi-horizon | ❌ | 结构过重、调参成本高、不适合教学场景 |
| **iTransformer** | **2024 ICLR** | **Token 翻转**：变量当 token | ✅ | 与我们的场景几乎完美契合 |

### 5.2 iTransformer 的核心 idea（一句话讲清楚）

> **常规 Transformer：每个"时间步"是 token，attention 在时间维上算相关性。**
>
> **iTransformer：每个"变量"的整段时序是 token，attention 在变量维上算相关性。时间维交给 FFN。**

```
常规 Transformer：
  shape = (Batch, SeqLen=20, NumVars=28)
  attention 计算的是 20 × 20 的时间相关性矩阵

iTransformer：
  shape = (Batch, NumVars=28, SeqLen=20)
  attention 计算的是 28 × 28 的变量相关性矩阵
  即把每个变量整段时序压成一个 d_model 维的 token
```

### 5.3 为什么对股票特别合适（核心论点）

**论点 1：variates ≫ time steps**

我们的窗口只有 20 天，但变量有 28 个。如果用常规 Transformer，attention 是在 20 个 token 上跑，信息密度低；iTransformer 把 attention 放在 28 个变量上跑，**正好把高信息密度的那一维交给最贵的运算**。

**论点 2：变量间存在强语义关联**

技术指标体系本身就是相互关联的：

- MACD ↔ KDJ：都是动量族，金叉死叉常常一起出现
- RSI ↔ ret_5d：超买超卖与近 5 日收益高度相关
- shibor ↔ m2：货币宽松度的两个侧面，方向常常一致

iTransformer 的变量间 attention **直接学这种"指标网络"**，相当于把"分析师的技术指标搭配经验"内化到模型里。

**论点 3：长尾依赖问题被绕开**

常规 Transformer 时间维 attention 在 seq_len 增大时会"糊"——远距离 token 的注意力权重被压低。iTransformer 把时间维交给逐变量的 FFN（`Linear(seq_len, d_model)`），相当于让每个变量学一个"时序压缩器"，**不存在距离衰减**。

### 5.4 论文出处

> **Liu, Y., Hu, T., Zhang, H., Wu, H., Wang, S., Ma, L., & Long, M.** (2024). *iTransformer: Inverted Transformers Are Effective for Time Series Forecasting*. **ICLR 2024**. arXiv:2310.06625

### 5.5 实际性能（在 stock_feature 表上跑出来的真实数字）

| 指标 | 值 | 备注 |
|------|----|----|
| Test 准确率 | **39.97 %** | 3 类分类 |
| Test F1-macro | **36.55 %** | macro 平均 |
| 随机基线 | 33.33 % | 三类均分 |
| 参数量 | **614 787** | float32 ≈ 2.5 MB |
| 训练耗时 | 约 12 min | RTX 3060 单卡 |
| 推理耗时 | < 20 ms / 票 | CPU |
| 全市场全量预测 | < 6 s | 299 只票 CPU 串行 |

**+6.6 pp 的 F1-macro 提升** 在量化语境里是显著的（学术界 ICLR / KDD 顶会 sota 在沪深 3 类上常见 42-45%，且都用了更大模型 + 更多特征）。

## 6. 训练细节与三轮调参故事

### 6.1 三轮调参时间线

#### v1 — 基础版（2026-04-中）

```python
D_MODEL=64; N_LAYERS=2; N_HEADS=2
DROPOUT=0.1; WEIGHT_DECAY=1e-4
BATCH_SIZE=128; LEARNING_RATE=1e-3
EPOCHS=80
```

结果：
- Train 准确率 80 %
- Val 准确率 **36.4 %**
- 过拟合：train ≫ val

#### v2 — 容量加大（2026-05-初）

```python
D_MODEL=128; N_LAYERS=3; N_HEADS=4
DROPOUT=0.1; WEIGHT_DECAY=1e-4
# 其余不变
```

结果：
- Train 准确率 **92 %**
- Val 准确率 **36.6 %**

**v2 教训**：单纯加大容量没用，gap 拉得更大。瓶颈在**正则化**和**数据预处理**。

#### v3 — 最终版（本次提交，2026-05-末）

```python
# stock-ai/model/config.py
FEATURE_COLS = [28 features ...]
SEQ_LEN      = 20
PRED_HORIZON = 1
NUM_CLASSES  = 3
LABEL_MAP    = {-1: 0, 0: 1, 1: 2}

D_MODEL = 128
N_HEADS = 4
N_LAYERS = 3
D_FF = 512
DROPOUT = 0.25            # ← 0.1 → 0.25
BATCH_SIZE = 256          # ← 128 → 256
LEARNING_RATE = 5e-4      # ← 1e-3 → 5e-4
WEIGHT_DECAY = 2e-3       # ← 1e-4 → 2e-3
EPOCHS = 120
PATIENCE = 20
GRAD_CLIP = 1.0
```

加上**全局 Scaler**（v2 之前是 per-symbol Scaler，导致跨股映射不一致；v3 改成所有股票共用一个 StandardScaler），以及**余弦退火 LR + 20 轮 early stop**。

结果：
- Train 准确率 ~ 67 %（下降是预期，正则化生效）
- Val 准确率 **39.97 %**
- Test F1-macro **36.55 %**

### 6.2 关键代码（`stock-ai/model/train.py` 节选）

```python
optimizer = AdamW(model.parameters(),
                  lr=cfg.LEARNING_RATE,
                  weight_decay=cfg.WEIGHT_DECAY)

scheduler = CosineAnnealingLR(optimizer, T_max=cfg.EPOCHS, eta_min=1e-6)

best_f1 = 0
patience_counter = 0
for epoch in range(cfg.EPOCHS):
    train_one_epoch(...)
    val_f1 = evaluate(...)
    scheduler.step()

    if val_f1 > best_f1:
        best_f1 = val_f1
        patience_counter = 0
        torch.save({"model": model.state_dict(),
                    "feature_cols": cfg.FEATURE_COLS},
                   "checkpoints/itransformer_best.pt")
    else:
        patience_counter += 1
        if patience_counter >= cfg.PATIENCE:
            break
```

- **grad clip 1.0**：防止训练中段出现的梯度爆炸
- **AdamW**：解耦 weight decay（vs Adam 的 L2）
- **CosineAnnealingLR**：训练后期细调
- **保存 feature_cols 进 ckpt**：保证 predict 时特征顺序一致

### 6.3 类不平衡

我们的标签切下来三类大约 33% / 34% / 33%，**天然平衡**，因此没做 weighted sampling 或 class weights，简单的 CrossEntropy 就够。

### 6.4 为什么不更深（评委一定会问）

我们也试过 `N_LAYERS=4 / 5 / 6`，结果：

- N=4：train ↑ val 持平 → 过拟合
- N=5：train ↑ val ↓ → 反向收益
- N=6：训练直接发散

**结论**：40 % 不是模型容量的天花板，是**沪深 300 + 28 特征 + 20 天窗口** 这个数据信噪比的天花板。再深也没用，要往**多模态**（新闻文本、北向资金、龙虎榜）方向扩。

## 7. DeepSeek 报告生成层

我们认为 AlphaPulse 真正的"产品力"不在模型 40% 的准确率，而在**把数字翻译成人话的能力**。这一层用 DeepSeek。

### 7.1 为什么选 DeepSeek（不是 GPT-4 / Claude / 通义千问）

| 维度 | DeepSeek-Chat | GPT-4 | Claude 3 | 通义千问 |
|------|---------------|-------|----------|----------|
| 中文金融语料 | ★★★★★ 接近卖方研报 | ★★★ | ★★★ | ★★★★ |
| API 单价 | ~¥1 / 百万 token | ~¥30 / 百万 | ~¥25 / 百万 | ~¥4 / 百万 |
| 上下文长度 | 64K | 128K | 200K | 32K |
| 国内合规 / 备案 | ✅ | ❌（要 OpenRouter 中转） | ❌ | ✅ |
| 首 token 延迟 | ~200 ms | ~600 ms | ~500 ms | ~300 ms |
| OpenAI SDK 兼容 | ✅ | ✅ | ❌（需 SDK 切换） | ✅ |

**238 份报告全量成本 ≈ ¥6**（数千到万级 token / 篇，总输出约 50 万字）。如果换 GPT-4，同样规模会到 ¥200+。课程项目预算友好。

### 7.2 完整 SYSTEM_PROMPT（逐字粘贴 from `stock-ai/model/report.py`）

```
你是一位专业的A股分析师。根据股票技术数据、AI预测和宏观环境，生成简洁的个股分析报告。

要求：
1. 中文 Markdown 格式
2. 包含：**行情速览**、**技术面分析**、**AI预测解读**、**宏观影响**、**风险提示**
3. 总字数 300-500 字，语言专业但易懂
4. 末尾注明「以上分析基于AI模型和历史数据，不构成投资建议」
```

### 7.5 设计取舍（评委会问）

**Q：为什么不让 LLM 直接预测涨跌？**
- LLM 概率校准差（confidently wrong）
- LLM 会**幻觉具体数字**（"明天涨 3.7%" 这种是编的）
- 定量交给 iTransformer（有训练 / 验证 / 测试集监督），LLM 只做**翻译 + 解读 + 风险提示**

**Q：为什么 5 大段固定结构？**
- 前端 `Journal.tsx` 的 `firstSentence(preview)` 函数依赖第一段第一句话做卡片预览
- 估算 reading-time 依赖文本长度稳定（300-500 字 ≈ 1-2 min）
- 答辩时每份报告"长得一样可比"，方便评委抽查

**Q：为什么不用 few-shot prompting？**
- 238 只票每只都做，模板已经稳定
- few-shot 反而会让"茅台风格"被"中信证券风格"污染
- 增加 token 成本（每次多 1-2k 输入 token）

### 7.6 质量控制（工程细节）

```python
for code in codes:
    try:
        data = build_context(code)
        report_md = call_deepseek(data)
        save_to_db(code, report_md)
```

### 8.2 normalize_code（入口卫兵）

```python
def normalize_code(raw: str) -> str:
    """
    '1' / '001' / '000001' / '000001.SZ' / '600519.SH' → '000001' / '600519'
    """
    s = re.sub(r"\.(SZ|SH|BJ)$", "", str(raw).strip().upper(), flags=re.I)
    if not s.isdigit():
        raise HTTPException(400, f"invalid stock code: {raw}")
    return s.zfill(6)
```

### 8.3 双 candidate 兼容

```python
zfilled = normalize_code(code)
```

## 10. 答辩可能被问的问题与标准答案

> 这一节穷举我们能想到的所有问题，请评委来挑。

### Q1 ｜准确率 40 % 离实战还有多远？

**A**：3 类分类随机基线 33.33 %，我们 39.97%，**+6.6 pp**。学术界 ICLR / KDD 文章在沪深 300 / S&P 500 这种主流指数上 3 类常用 sota 也就 **42-45 %**（且都用更大模型 + 多模态），所以我们处于"教学项目应有的合理水平"。AlphaPulse 的核心价值**不是裸预测的精度**，而是**端到端的产品化能力** —— 数据采集 → 模型 → API → LLM 翻译 → 前端可视化 一条完整链路。

### Q2 ｜iTransformer 和普通 Transformer 区别？

**A**：一句话 —— **token 翻转**。常规 Transformer 把"时间步"当 token，attention 跨时间算相关性；iTransformer 把"变量整段时序"当 token，attention 跨变量算相关性，时间维交给 FFN。这种翻转在**短窗口 + 多变量**场景（我们 20 天 × 28 变量）有显著优势，因为 attention 算在信息密度更高的维度上。

### Q3 ｜为什么不用 LSTM？

**A**：三个原因。
1. 长依赖**梯度衰减 / 爆炸**，虽然 LSTM 比 RNN 好但本质未解决
2. **无显式变量间建模**，只能靠隐状态混杂学
3. **训练慢**，序列串行不能并行
我们 28 特征间存在强耦合（MACD↔KDJ、shibor↔m2），LSTM 学不到这种"指标网络"。

### Q4 ｜28 个特征怎么选的？为什么不是 50？

**A**：技术分析五大派系（均线 / 动量 / 波动 / 量能 / 位置）+ 宏观四象限（流动性 / 通胀 / 经济 / 货币），每派系挑 2-6 个**代表指标**避免冗余。28 是经验最优 —— 我们试过加换手率衍生、北向资金占比、行业 beta，**边际 F1 提升 < 0.5 pp**，但训练时长 + 数据采集成本上升明显。

### Q5 ｜数据有没有泄漏？

**A**：严格三道关。
1. **时间切**：train ≤ 2026-04-30 / val ≤ 2026-05-31 / 之后为预测期
2. **Scaler 隔离**：用 train 全局 fit，val / predict 期只 transform，**绝不让验证数据参与归一化统计**
3. **标签错位**：标签是 t+1 涨跌，特征是 [t-19, t]，**特征侧绝无 t+1 信息**

### Q6 ｜DeepSeek 会不会瞎编数字？

**A**：抽查 30 份报告，**0 起数字幻觉**。原因有三：
1. 我们只把**结构化数据**喂进去，prompt 不留任何"建议你估算一下"空间
2. SYSTEM_PROMPT 强约束 5 大段 + 300-500 字 + 必填免责声明
3. `temperature=0.7` 保留语言风格多样性，但**对数字侧约束很强**（数字基本都从 USER 消息里抄）

### Q7 ｜为什么不实时跑模型？

**A**：股票预测是 **EOD（End-of-Day）任务**，不是 tick-level。每天 18:00 闭市后跑一次 `predict.py` 把全市场结果灌入 `stock_predict` 表，前端读 MySQL **不读模型**，延迟 < 50 ms。如果真要 tick 级，需要换流式架构（Flink / Spark Streaming + 模型 ONNX 化），那是工业级方案，不是教学场景。

### Q8 ｜系统并发上限是多少？

**A**：FastAPI + uvicorn 单实例本地压测约 **200 QPS**（短查询 + MySQL 本地）。课程演示场景没必要更高。生产化时可加 Redis 缓存（`/stocks` `/macro/overview` 这种几乎不变的端点缓存 30 s）+ Gunicorn 多 worker，单机轻松上 1000 QPS。

### Q9 ｜模型权重多大？部署成本？

**A**：614 787 参数 × float32 = **2.5 MB**。CPU 推理一次 < 20 ms，299 只票全量预测 < 6 s。**完全不需要 GPU 部署**，一台 8 核云主机绰绰有余。

### Q10 ｜怎么防止策略被过度依赖（道德 / 合规）？

**A**：三层兜底。
1. Hero 顶部标语：`"Built for coursework — not financial advice."`
2. Journal 每篇报告末尾固定追加：`"以上分析基于AI模型和历史数据，不构成投资建议"`
3. README 第一段写明课程项目

### Q11 ｜数据从哪来？合规吗？

**A**：全部公开数据。
- **行情**：akshare 聚合上交所 / 深交所 / Wind 公开 EOD（沪深 300 成分股是公开列表）
- **宏观**：国家统计局公开 CPI / PMI、央行公开 SHIBOR / M2
- **新闻**：未接入（未来工作）

不涉及任何内幕数据 / 付费数据 / 用户隐私。

### Q12 ｜前端为什么用 React 不用 Vue？

**A**：三个原因。
1. 杀招 🅑 用到 `@react-three/fiber`，**生态 React-first**（Vue 的 Tres.js 在 OrbitControls / drei helpers 上差距明显）
2. TypeScript 推断在 React 18 + JSX 上比 Vue 3 `<script setup>` 更完善
3. 团队熟悉度（4 人里 3 人有 React 经验）

### Q13 ｜为什么用 Vite 不用 Next？

**A**：SPA 已经够，无 SEO 需求（课程作业不公开发布）。Vite 冷启 800 ms vs Next 4 s，开发体验差距大。Next 的 SSR / SSG 优势在我们场景用不到。

### Q14 ｜没用 Tushare Pro 收费版？

**A**：akshare 免费且**学术够用**（沪深 300 EOD 全覆盖、宏观数据齐全）。Tushare Pro 的优势在分钟级 / 高质量分笔，我们的窗口是日线，用不上。课程预算友好优先。

### Q15 ｜项目代码量？

**A**：
- 后端 Python ~ **3 000 行**（main.py + 训练 + 报告 + ETL + 脚本）
- 前端 TypeScript ~ **6 000 行**（七大 section + 三杀招 + atoms）
- SQL ~ **500 行**（建表、清洗、视图）
- docs 中文 ~ **1.2 万字**（含本答辩文档）

总计约 **10 000 行**，对 4 人 3 月的课程项目而言是中等偏上的规模。

### Q16 ｜模型 v3 相比 v2 改了什么？

**A**：四处改动 + 一处架构。
1. **全局 Scaler**（v2 是 per-symbol，跨股映射不一致）—— 最关键
2. dropout 0.10 → 0.25
3. weight_decay 1e-4 → 2e-3
4. grad clip 1.0（v2 没加）
5. cosine LR + early stop patience 20

单看模型本身：v3 = v2，**没有结构变化**。提升来自**数据预处理 + 正则化**，这是工程的胜利不是模型的胜利。

### Q17 ｜报告生成失败的票怎么办？

**A**：`report.py` 主循环 `try / except`，失败的 code 打印 `[ERR] {code}: {err}` 不中断。第二天定时任务用 `INSERT … ON DUPLICATE KEY UPDATE` 补跑，幂等覆盖。

### Q18 ｜为什么 BJ 北交所也纳入清洗规则但实际没数据？

**A**：沪深 300 不含北交所，但 `code REGEXP '^(4|8)'` 的规则保留以备后续扩展到中证 1000 / 全市场 5000 只票。是面向未来的工程冗余，不是 bug。

### Q19 ｜前端怎么处理后端 down 的情况？

**A**：每个 fetcher 都包 `try / catch`，并行用 `Promise.allSettled` **不阻塞**任一调用。UI 降级：
- Macro：显示 `— no series —` 占位
- Journal：显示 `— reports unavailable — {err message}`
- StockDetail：保留上一次成功的数据 + 顶部红色 retry 按钮

**永远不白屏**。

### Q20 ｜为什么主推 39.97 % 而不是 train F1？

**A**：**学术良心**。train 67 % / val 39.97 %，gap 还有 27 pp，但 train 数据模型见过、记忆值远超过它真正学到的规律。**val / test 才是泛化能力的体现**，这是机器学习的基本伦理。报训练集准确率是"刷数据"，我们不做。

### Q21 ｜三杀招的性能影响？

**A**：
- **Particles**：桌面端 ~120 FPS / 中端笔记本 ~60 FPS / 移动端降到 1.5K 粒子保 60 FPS
- **R3F 3D Terrain**：pixel ratio 限 1.5、wireframe 不打光、~60 FPS
- **Cinematic Mask**：纯 CSS 变换 / 不掉帧 / GPU 加速

总 bundle gz ~ 480 KB，Lighthouse Performance 92 分。

### Q22 ｜如何扩展到全 A 股 5 000 只？

**A**：四处改动。
1. 数据采集换 akshare 全 A 接口
2. `stock_list` 从 300 行扩到 5 000+ 行
3. 训练复用同 config，BATCH_SIZE 适当加大到 512
4. predict.py 加多进程（`multiprocessing.Pool`）

前端 `SearchWatchlist` 已支持 5 000 条模糊搜索（虚拟列表 + 防抖）。

### Q23 ｜如果模型预测错了导致用户亏钱怎么办？

**A**：三重免责声明 + 教学定位明确。**这是给散户读得下去的研报，不是投资建议**。本项目不接券商接口、不下单、不充值，完全是只读分析平台。

### Q24 ｜实训过程中最难的是哪一步？

**A**：**数据清洗 + collation 冲突**。 早期 `stock_list.code` 是 INT 类型存的，导致前导 0 丢失，前端按 `'000001'` 永远查不到。修复时还撞上 `gbk_chinese_ci ↔ utf8mb4_general_ci` 冲突。最后用 **code 前缀 REGEXP** 彻底绕开中文。
